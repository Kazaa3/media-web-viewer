# Video Player Architecture Overview

## Introduction
The Video Player in the Media Web Viewer is a multi-engine playback system designed to handle a wide range of media formats by dynamically selecting the best playback strategy (Native Browser vs. VLC Fallback vs. Fragmented Transcoding).

## Playback Orchestration

### 1. Media Analysis & Routing
When a media item is selected, the `playVideo(item, path)` function in `app.html` triggers a two-step process:
1.  **Backend Analysis**: `eel.analyze_media(path)` is called to determine the media's technical properties (codecs, HDR, resolution).
2.  **Routing Decision**: Based on the analysis, a `mode` is selected:
    -   **`direct`**: For browser-native formats (.mp4, .webm). Served via `/direct/` route with byte-range support.
    -   **`transcode`**: For incompatible formats or disc images (.iso). Triggers on-the-fly FFmpeg transcoding to Fragmented MP4 (fMP4).
    -   **`hls`**: For adaptive streaming or complex remuxing.
    -   **`vlc_embedded`**: Fallback for formats that the browser cannot handle even with transcoding (e.g., complex DVD menus or high-bitrate HDR that Chrome struggles with).

### 2. Frontend Player (Video.js)
The frontend uses **Video.js** as the primary orchestration layer. It handles:
-   **UI Controls**: Time-seeking, volume, fullscreen, and PiP.
-   **Event Loop**: Syncing duration and current time back to the UI.
-   **Error Handling**: Automatically detecting when a source fails and offering fallback hints.
-   **Stream Seeking**: For fragmented streams, seeking triggers a reload with a start-time offset (`?ss=XX`) to the backend.

## Backend Routes (Bottle)

-   `/direct/<path>`: Serves raw files using `static_file`. Crucial for native browser seeking (requires `Range` header support).
-   `/video-stream/<path>`: FFmpeg-backed fMP4 streaming.
-   `/vlc-stream/<item_id>`: Legacy pipe-based streaming for VLC.
-   `/vlc-hls-live/<path>`: HLS stream generated by a background VLC process.

## Architectural Patterns
-   **Handler Pattern**: `VideoHandler` and `AudioHandler` isolate the logic for different media types, resolving paths and determining playback modes before they reach the main orchestrator.
-   **Pipeline Integration**: A dedicated metadata pipeline ensures that technical properties are extracted once and cached in the database, enabling fast routing decisions.
-   **Async Handover**: Eel is used to bridge the Python backend and HTML/JS frontend, with events like `on_playback_start` and `on_playback_error` providing a reactive UX.
